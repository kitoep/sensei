# Table design — intake, types, and naming

**No CREATE TABLE without intake.** The schema is derived from the queries and the data's lifecycle, not the other way around.

## Minimum intake (answer or ask BEFORE designing)

| # | Question | Why it changes the design |
|---|---|---|
| 1 | What queries will read this table? (the top 3-5, with their filters and ordering) | Drives indexes, denormalization, and whether JSON is acceptable. |
| 2 | Cardinalities? (1:1, 1:N, N:M; how many children per parent typically?) | Drives FKs, join tables, and whether embedding makes sense. |
| 3 | Expected volume at 1-2 years? (hundreds, millions, hundreds of millions of rows?) | Drives ID types, partitioning, and which migrations will be possible later. |
| 4 | Is it multi-tenant? What's the tenant key? | Every tenant table carries `tenant_id`/`organization_id` NOT NULL + index; for RLS and query-level isolation, see the `security-compliance` skill if installed. |
| 5 | Lifecycle? (is it updated, append-only, deleted, how long is it retained?) | Drives soft-delete, history/event tables, purge policies. |
| 6 | Read/write ratio? (a dashboard that reads a lot, or ingestion that writes a lot?) | Drives how many indexes you can afford. |
| 7 | Who else writes/reads the DB? (another app, ETL, humans with SQL) | If the app isn't the only client, app validation protects nothing: constraints are mandatory. |

If the user doesn't know an answer, write the explicit assumption into the plan ("assuming <1M rows/year") — a written assumption can be corrected; an implicit one can't.

## Type table — exact rules (Postgres; MySQL note where it differs)

| Data | Correct type | FORBIDDEN | Note |
|---|---|---|---|
| Money | `NUMERIC(12,2)` or `BIGINT` cents | `FLOAT`/`REAL`/`DOUBLE` | Float accumulates rounding error: 0.1 + 0.2 ≠ 0.3. Use cents if you do heavy arithmetic. |
| Event timestamp | `TIMESTAMPTZ` | `TIMESTAMP` without tz, `VARCHAR` | Always with timezone; store in UTC, convert on display. MySQL: `TIMESTAMP` (converts to UTC) or `DATETIME` + documented UTC convention. |
| Date without time (birthday, cutoff date) | `DATE` | `TIMESTAMPTZ` | A birthday has no timezone. |
| Status enum | `TEXT` + `CHECK (status IN (...))`, or lookup table if it grows | free `VARCHAR`, magic integers (1=active) | Postgres's native `ENUM` type is hard to alter; CHECK is just as strong and migratable. |
| Boolean | `BOOLEAN NOT NULL DEFAULT ...` | `VARCHAR('yes'/'no')`, `INT` 0/1 | Always with a default: a NULL boolean is an accidental third state. |
| Primary ID | Conscious decision: `BIGINT GENERATED ALWAYS AS IDENTITY` or `UUID` (v7 if available) | 32-bit `SERIAL` on large tables, UUID v4 as PK on high-write tables | Identity/bigint: compact, ordered, enumerable (count leak if public). UUID: not enumerable, client-generatable; v4 fragments indexes, v7 doesn't. If the ID travels in public URLs → UUID or a separate public id. |
| Phone, zip, tax id, external ref | `TEXT`/`VARCHAR` | `INT`/`BIGINT` | "Numbers" you don't add or subtract are text: they preserve leading zeros and formatting. |
| Quantities/counters | `INT`/`BIGINT` with `CHECK (>= 0)` where applicable | — | Think about the range: `INT` overflows at 2.1B. |
| Free text | `TEXT` | `VARCHAR(255)` out of habit | In Postgres `VARCHAR(n)` saves nothing; cap it only if there's a real business rule (and with CHECK). MySQL: `VARCHAR` with a limit does matter for indexes. |

## JSON vs columns — decision rule

A `JSONB` column is valid ONLY if the data meets all three:
1. You don't filter or order by it in frequent queries (or only by keys you can index with GIN, and you measured it).
2. It has no FK to other tables.
3. Its structure varies per row (webhook payloads, heterogeneous product attributes, raw LLM responses).

Everything else — status, amounts, dates, references — is a typed column with a constraint. "I'll stuff it in JSON to avoid a migration" = debt you'll pay with a much worse backfill migration later.

## Naming — fixed convention

- Tables: plural, `snake_case` (`orders`, `order_items`). Pick singular or plural once per project and DON'T mix; if the project already has a convention, follow it.
- Columns: `snake_case`; FKs as `<singular>_id` (`order_id`, `tenant_id`).
- Booleans: `is_`/`has_` prefix (`is_active`, `has_invoice`).
- Event timestamps: `_at` suffix (`created_at`, `paid_at`, `deleted_at`); dates without time: `_on` or `_date`.
- Indexes/constraints: name them explicitly (`idx_orders_tenant_created`, `uq_users_email`, `fk_items_order`) — autogenerated names make errors and drop migrations unreadable.

## Mandatory columns on every business table

```sql
id            BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,  -- or UUID: intake decision
tenant_id     BIGINT NOT NULL REFERENCES tenants(id),           -- if multi-tenant, ALWAYS
created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()                -- updated by trigger or ORM
```

## Soft-delete — explicit decision, not a default

- Use `deleted_at TIMESTAMPTZ NULL` (not `is_deleted`: the timestamp tells you when, and NULL = alive).
- Real cost: ALL queries must filter `WHERE deleted_at IS NULL` (global ORM scope) and UNIQUEs must be partial: `CREATE UNIQUE INDEX uq_users_email ON users(email) WHERE deleted_at IS NULL;` — otherwise you can't recreate a deleted user.
- Hard-delete is correct when there's a legal obligation to delete (GDPR/CCPA) or the data has no historical value. Decide per table and write it down.

## History and auditing

If the intake says "we need to know who changed what" or the data is a transitioning state (orders, tickets): an append-only event table (`order_events(order_id, event_type, payload jsonb, actor_id, created_at)`) alongside the current state. Reconstructing history from `updated_at` is impossible; adding it later requires that the past already be lost.
